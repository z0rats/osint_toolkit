# OSINT Toolkit ā€” project context

Self-hostable, single-user OSINT/security-analyst web app. FastAPI backend + React frontend, runs via Docker Compose. Not production-hardened (early prototype, per README).

**Maintenance:** update this file after any important project change (new feature, new external dependency, architectural shift). Keep it short and specific ā€” no speculative or redundant detail.

## Stack

- **Backend**: Python 3.14, FastAPI, SQLAlchemy 2.0 (async, `aiosqlite`/`asyncpg`), Alembic migrations, APScheduler for background jobs, `pydantic-ai` for LLM features (Anthropic/OpenAI/Google/Groq/Mistral/xAI/Cohere/Bedrock all wired via `pydantic-ai-slim` extras), `slowapi` for rate limiting, pytest for tests.
- **Frontend**: React 19, MUI 9, Jotai (state), react-router-dom 7, react-scripts 5 (CRA, `--openssl-legacy-provider`), Yarn 4 (berry, `packageManager: yarn@4.15.0`).
- **DB**: SQLite by default at `data/osint_toolkit.db` (see `backend/app/core/config/settings.py` ā€” `DatabaseSettings`, env prefix `DB_`). Logs at `data/logs/`.
- **Deploy**: `docker-compose.yaml` ā€” `backend` (no exposed port, mounts `./data`) + `frontend` (nginx, port 4000). `make up` / `make rebuild` / `make up-backend` / `make up-frontend` (see `Makefile`). Backend runs as non-root `appuser` (UID/GID 1000): `backend/docker-entrypoint.py` starts as root just long enough to `chown` the bind-mounted `./data` (host-owned, so it can't be fixed at image-build time), then drops to `appuser` via `setuid`/`setgid` and `execvp`s into the real `uvicorn` command ā€” no `gosu`/`su-exec` needed. `appuser`'s home directory (created at build time via `useradd -m`, not bind-mounted) is where maigret's site-database auto-update (`~/.maigret`) and pyppeteer's lazily-downloaded Chromium (`~/.local/share/pyppeteer`, only if `email_search`'s headless checks are enabled) end up.

## Backend architecture

Entry point: `backend/main.py` ā€” builds the FastAPI app via `create_fastapi_application()`, lifespan hooks create DB tables, run app defaults, start background favicon fetch, start scheduler.

Layout under `backend/app/`:
- `core/` ā€” cross-cutting: `config/` (settings, CORS, security headers, rate limiting, request-id, body-limit middleware), `settings/` (persisted app settings: `api_keys` ā€” `Apikey.key` is encrypted at rest via `EncryptedString`/`secrets_crypto.py` (Fernet; key from `SECURITY_ENCRYPTION_KEY` or auto-generated at `<data_dir>/.encryption_key`, 0600 ā€” back that file up with the DB, or losing it makes stored keys unrecoverable; legacy plaintext rows still read fine and get encrypted on next save), `ai_settings`, `general`, `keywords`, `modules`, `cti_profile` ā€” each has its own models/schemas/crud/routers under `core/settings/<name>/`), `security/ssrf_guard.py` (resolves + validates the hostname of any user-supplied URL before it's fetched server-side, rejecting private/loopback/link-local/reserved IPs; `safe_get(client, url, ...)` is the single shared entrypoint ā€” pins the resolved IP for the request *and* manually re-validates each redirect hop, since httpx's own `follow_redirects` would re-resolve `Location` without going through this check; callers must construct their `httpx.AsyncClient` with `follow_redirects=False`. Used by `favicon_downloader.py`, `newsfeed/utils/fetching.py` (fetches article/feed bytes itself and feeds them to `newspaper`/`feedparser` instead of letting those libraries make their own unguarded request), and `llm_templates/utils/web_content_fetcher.py`; opt-out via `SECURITY_ALLOW_PRIVATE_NETWORK_TARGETS`, dev/testing only), `reports/` (generic HTML/PDF report renderer ā€” `render_html`/`render_pdf` via Jinja2 + `xhtml2pdf`, `generate_report(title, sections, fmt, locale, ...)`; feature-level `report_service.py` modules build `ReportSection`/`ReportRow` lists with their own small `en`/`ru` label dicts and call into this; used by `ioc_lookup` single-lookup history (`GET /api/ioc-lookup/history/{id}/report`) and `email_analyzer` (`POST /api/email/report`, stateless ā€” takes the analysis result in the body since there's no persisted history to fetch by ID; `username_search`'s own export is unrelated ā€” it reuses Maigret's native report generators directly, not this layer), `alerts/` (WebSocket alerts), `database.py`, `scheduler.py`, `dependencies.py`, `exceptions.py`, `healthcheck.py`.
- `features/` ā€” one directory per product feature, each typically split into `routers/`, `service/`, `schemas/`, `models/`, `crud/`, `utils/`, `config/`:
  - `newsfeed/` ā€” RSS aggregation, IOC extraction from articles, MITRE ATT&CK mapping, trends/analytics.
  - `ioc_tools/` ā€” `ioc_lookup` (single + bulk lookups against AbuseIPDB, AlienVault, VirusTotal, Shodan, etc.; single-lookup searches are auto-saved to history once every queried service responds, via `POST/GET/DELETE /api/ioc-lookup/history`, persisted as `SingleLookupSearch`/`SingleLookupResult`), `ioc_extractor`, `ioc_defanger`, `domain_finder` (URLScan.io-based typosquat/phishing domain discovery).
  - `email_analyzer/` ā€” `.eml` parsing, header/IOC checks, AI-assisted analysis.
  - `image_tools/` ā€” EXIF/metadata extraction, hashing, reverse image search deep-links (no API key needed).
  - `llm_templates/` ā€” user-defined AI prompt templates (categories + templates).
  - `cvss_calculator/` ā€” CVSS 3.1/4.0 scoring.
  - `username_search/` ā€” two pluggable sources behind one unified schema (`MaigretSearch`/`MaigretSiteResult`, with a `source` column: `maigret` | `social_analyzer`, and a `extra` JSON column for source-specific display data), selected per-scan via `ScanRequest.source` (`POST /api/username-search/scan`):
    - **maigret** ā€” called in-process (not subprocess) via `maigret.checking.maigret()`, true per-site progress streamed over SSE; settings under `core/settings/username_search/` (timeout/concurrency/top-sites-count/proxy, plus site-database auto-update tracking).
    - **social-analyzer** ā€” invoked as a **subprocess** of its installed CLI (`social_analyzer_service.py`, `--output json`), *not* in-process: the pip package's on-disk module directory is literally named `social-analyzer` (hyphen), which isn't a valid Python identifier, so `import social_analyzer` doesn't work ā€” only `importlib.metadata` (version/site-count) and the console script are usable. No per-site progress callback exists either, so the SSE stream is coarse (`started` ā†’ one terminal event); cancellation kills the subprocess. Settings/version-check under `core/settings/username_search/social_analyzer_settings_*` (timeout/top-sites-count, plus a manual PyPI-latest-version check ā€” installing a newer version still requires a container rebuild since site data ships inside the pip package, there's no separate remote DB updater like Maigret's).
    - Report export (`report_service.py`) reuses Maigret's own report writers and only works for maigret-sourced runs; social-analyzer runs just report `has_export: false`.
  - `email_search/` ā€” finds which mail providers a username is registered at, via `mailcat-osint`'s ~26 per-provider checker coroutines (SMTP RCPT probing, provider APIs, registration-form probing, headless Chromium). Driven in-process (no reusable entrypoint exists in mailcat like maigret's, so `service/email_search_service.py` runs the checker coroutines itself, bounded by an `asyncio.Semaphore`), progress streamed over SSE (`POST /api/email-search/scan`), found-provider results persisted (`MailSearch`/`MailSearchResult`). Two checker groups are gated off by default via settings (`core/settings/email_search/`, `enable_smtp_checks`/`enable_headless_checks`): SMTP checkers (Gmail/Yandex/mail.de) need outbound TCP/25 usually blocked in Docker/cloud (workaround: `use_tor`/`proxy_url`, mirroring mailcat's own `--tor`/`--proxy`); headless-Chromium checkers (Fastmail/int.pl/onet.pl, via `requests-html`ā†’`pyppeteer`) lazily download a Chromium binary on first real use. Like social-analyzer, version/update-check is a manual PyPI-latest check (`GET /api/email-search/info`, `POST /api/email-search/check-update`) ā€” installed version read from package metadata (no `__version__` attribute on the `mailcat` module itself), latest-available version persisted on `EmailSearchConfig`; installing a newer version still requires a container rebuild.
- **Access control**: the app has no user accounts (single-user tool) ā€” instead, `core/security/access_control.py` guards every `/api/*` route (plus `/docs`/`/redoc`/`/openapi.json`, added back in `main.py` after `fastapi_config.py` disables FastAPI's built-in ones) behind one shared bearer token, checked via `verify_access_token` wired as a router-level dependency in `router_registry.py`. Token comes from `API_ACCESS_TOKEN` or is auto-generated and persisted to `<data_dir>/.access_token` (0600) on first startup, printed once to stdout/logs ā€” same pattern as the encryption key, factored into `core/security/persisted_secret.py` and shared by both. `healthcheck.router` and the alerts `ws_router` are the only routers registered without this dependency: healthcheck has no sensitive data, and the WebSocket handshake checks the same token itself via a `?token=` query param (browsers can't set a custom header on a WS handshake) ā€” see `alerts_routes.py`. Frontend: `core/services/baseApi.js` attaches the token from `localStorage` (`core/utils/accessToken.js`) to every request and reloads the page on a 401 from an already-set token; `core/components/ui/AccessGate.jsx` shows a one-time entry screen when no token is stored yet, wrapping `AppRoutes` in `App.js`.
- All routers are aggregated in `backend/app/utils/router_registry.py` via `get_core_routers()`, `get_settings_routers()`, `get_feature_routers()`, then mounted in `main.py`.
- Migrations: Alembic, config at `backend/alembic.ini`, scripts in `backend/migrations/versions/`. Run after schema changes: `docker compose run --rm backend alembic upgrade head`.
- Tests: `backend/tests/`, run with pytest (`pytest.ini` sets `testpaths = tests`, `pythonpath = .`). Currently sparse coverage (only `tests/features/image_tools`).
- CI (`.github/workflows/ci.yml`): besides backend/frontend tests and a docker-build smoke test, runs `pip-audit`/`yarn npm audit` (dependency CVEs) and Trivy (built-image CVEs) as report-only (`continue-on-error: true` ā€” this dependency tree pulls in several OSINT tools outside this repo's control, e.g. maigret/social-analyzer/mailcat-osint/pyppeteer, likely already carrying unfixable transitive findings), plus CodeQL (SAST, Python+JS) and gitleaks (secret scanning) as separate blocking jobs. All four are free for this public repo ā€” gitleaks-action specifically requires a paid license only for org-owned repos, not personal ones.
- `.github/dependabot.yml` opens weekly update PRs for `pip` (backend), `npm-and-yarn` (frontend), `docker` (both Dockerfiles), and `github-actions` (the CI workflow itself).
- `.pre-commit-config.yaml` runs gitleaks locally pre-commit (same tool as CI's `secret-scan` job, but catches a secret before it ever enters git history rather than after a push). One-time setup per clone: `pip install pre-commit && pre-commit install`.
- `docker-compose.yaml`'s `backend` service has `env_file: {path: .env, required: false}` ā€” a root-level `.env` (see `.env.example`) is NOT picked up by the container otherwise; compose's own automatic `.env` loading only covers variable substitution *within* the compose file, not the container's process environment.

## Frontend architecture

`frontend/src/features/` mirrors the backend feature split ā€” one directory per module (`newsfeed`, `ioc-tools`, `email-analyzer`, `image-tools`, `llm-templates`, `cvss-calculator`, `rule-creator` (Sigma/Yara/Snort rule builder), `username-search`, `email-search`, `settings`). Each feature dir typically has its own `components/`, `hooks/`, `services/` (API calls), `constants/`, `utils/`. Entry: `src/App.js` ā†’ `src/index.js`. `src/core/` holds shared/cross-feature code.

New top-level features need edits in three places beyond their own directory: `src/core/config/routes.js` (lazy route), `src/core/config/sidebarConfig.js` (nav entry + tabs config), and `src/core/components/layout/Layout.jsx` (`currentTabs` path match for the new tabs getter). i18n: add a `<featureName>.json` under both `src/core/i18n/locales/en/` and `ru/`, register in `src/core/i18n/index.js`, and add `nav.*` keys to `common.json` (both locales).

## Integrated external services

IOC/threat-intel lookups across IPs, domains, URLs, emails, hashes, CVEs via: AbuseIPDB, AlienVault OTX, CheckPhish, CrowdSec, GitHub, Google Safe Browsing, HIBP, Hunter.io, IPQualityScore, Maltiverse, NIST NVD, Pulsedive, Reddit, Shodan, ThreatFox, Twitter/X, URLScan.io, VirusTotal. ## Conventions worth knowing

- Settings/config uses `pydantic-settings` with per-domain `BaseSettings` subclasses, each with its own `env_prefix` (e.g. `DB_`, `LOG_`, `API_`, `SCHEDULER_`), all loaded from a single `.env` (gitignored, not present in repo ā€” none committed).
- `core/config/log_redaction.py`'s `SecretRedactionFilter` is attached to every logging handler (console + both file handlers, `logging_config.py`) and scrubs API-key-shaped substrings (query params, `Bearer` tokens, IPQualityScore's path-embedded key) out of every log record. Needed because several IOC-lookup providers (Hunter.io, Pulsedive, Shodan, Google Safe Browsing, IPQualityScore) pass their key in the request URL rather than a header, and `httpx.HTTPStatusError`'s `str()` embeds the full URL ā€” a broad `except Exception: logger.error(..., str(e))` would otherwise leak it into `data/logs/*.log`. Matches by pattern (not by looking up live key values), so new providers are covered automatically.
- In route handlers, prefer injecting settings via dependency (`SettingsDep`) rather than importing the module-level `settings` singleton, so tests can override via `app.dependency_overrides`.
- Feature module layering pattern: `routers` (HTTP) ā†’ `service` (business logic) ā†’ `crud` (DB access) ā†’ `models`/`schemas`. New features should follow this same split.
- AGPL-3.0 licensed (versions ā‰¤ v0.1.0 are MIT). Be mindful of this when adding dependencies or discussing licensing.
- Any code that makes an HTTP request to a URL/host supplied (directly or indirectly) by the user must go through `core/security/ssrf_guard.py` first (see `favicon_downloader.py`) ā€” otherwise it's an SSRF vector into the docker network, localhost, or cloud metadata endpoints.
- Security headers are set in two independent places, since the backend's middleware never sees the SPA's own response: `core/config/security_config.py`'s `SecurityHeadersMiddleware` covers `/api/*`, `/docs`, `/redoc`, `/openapi.json` (CSP is relaxed to allow `cdn.jsdelivr.net` + inline init script specifically on `/docs`/`/redoc`, since Swagger/ReDoc need it and those routes are access-token gated with no user-supplied content anyway); `frontend/nginx.conf`'s `location /` sets the same base headers + a strict CSP for the actual HTML/JS the browser renders. HSTS stays production-only (`ENVIRONMENT=production`) since it must not be sent over plain HTTP, which is this app's default.
- `backend/requirements.txt` is compiled (pip-compile/`uv pip compile` style, hand-merged since no `requirements.in` is committed) ā€” `maigret` declares `lxml>=6.0.2` but `newspaper4k` pins `lxml<6.0.0`; resolved via `backend/lxml-override.txt` (`lxml==5.4.0`, passed as `--override` in `Dockerfile`) since maigret doesn't actually import lxml directly. Don't bump lxml without re-checking this (`social-analyzer` also declares an unpinned `lxml` dependency but likewise never imports it directly ā€” verified compatible with the same override; `mailcat-osint`'s `requests-html`ā†’`pyppeteer` chain was also verified compatible with the same override, no additional pin needed).

## Useful commands

- `make up` / `make rebuild` ā€” start (or rebuild + start) full stack via Docker.
- `make up-backend` / `make rebuild-backend`, `make up-frontend` / `make rebuild-frontend` ā€” per-service.
- `docker compose run --rm backend alembic upgrade head` ā€” apply DB migrations.
- Backend tests: `cd backend && pytest`.
- Frontend dev: `cd frontend && yarn start` (uses `--openssl-legacy-provider`); build: `yarn build`; test: `yarn test`.
- App served at `http://localhost:4000` once containers are up.
