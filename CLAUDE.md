# Corvid — project context

Self-hostable, single-user OSINT/security-analyst web app. FastAPI backend + React frontend, runs via Docker Compose. Not production-hardened (early prototype, per README).

**Maintenance:** this file loads in full on every message regardless of task, so its size is a standing tax on every conversation — treat growth as a cost, not a free action.

- Update it after any important project change (new feature, new external dependency, architectural shift) — but each edit is a *net* pass: prune or tighten something (merge, generalize, cut what's now covered elsewhere or obvious from the code) roughly as much as you add. Don't just append.
- Describe the **current state only**. No "was X, changed to Y because Z", no "see commit/PR #123", no incident narratives, no migration history — that belongs in commit messages and git history, which are queryable on demand and don't cost tokens on every turn. If a past decision needs to stay discoverable, a short present-tense reason clause (e.g. "X, since Y" ) is fine; a paragraph of backstory is not.
- No speculative or redundant detail.
- If this file is pushing 20KB+, that's a signal to condense before adding more — collapse verbose bullets, drop detail that's redundant with the code itself, and prefer pointing at a file/symbol over re-explaining what it does.
- Also periodically update `README.md` with important user-facing changes — especially anything affecting deployment (new required setup step, changed env vars, new `.env`/`docker-compose.yaml` behavior) — since that's what operators actually read when installing/upgrading.

## Stack

- **Backend**: Python 3.14, FastAPI, SQLAlchemy 2.0 (async, `aiosqlite`/`asyncpg`), Alembic migrations, APScheduler for background jobs, `pydantic-ai` for LLM features (Anthropic/OpenAI/Google/Groq/Mistral/xAI/Cohere/Bedrock all wired via `pydantic-ai-slim` extras), `slowapi` for rate limiting, pytest for tests.
- **Frontend**: React 19, MUI 9, Jotai (state), react-router-dom 7, react-scripts 5 (CRA, `--openssl-legacy-provider`), Yarn 4 (berry, `packageManager: yarn@4.15.0`).
- **DB**: SQLite by default at `data/corvid.db` (see `backend/app/core/config/settings.py` — `DatabaseSettings`, env prefix `DB_`). Logs at `data/logs/`.
- **Deploy**: `docker-compose.yaml` — `backend` (no exposed port, mounts `./data`) + `frontend` (nginx, port 4000). `make up` / `make rebuild` / `make up-backend` / `make up-frontend` (see `Makefile`). Backend runs as non-root `appuser` (UID/GID 1000): `backend/docker-entrypoint.py` starts as root just long enough to `chown` the host-owned bind-mounted `./data`, then drops to `appuser` via `setuid`/`setgid`, runs `alembic upgrade head`, and `execvp`s into the real `uvicorn` command. `appuser`'s home directory (build-time, not bind-mounted) is where maigret's site-database (`~/.maigret`) and pyppeteer's lazily-downloaded Chromium (`~/.local/share/pyppeteer`, only if `email_search`'s headless checks are enabled) end up.

## Backend architecture

Entry point: `backend/main.py` — builds the FastAPI app via `create_fastapi_application()`, lifespan hooks create DB tables, run app defaults, start background favicon fetch, start scheduler.

Layout under `backend/app/`:
- `core/` — cross-cutting: `config/` (settings, CORS, security headers, rate limiting, request-id, body-limit middleware), `settings/` (persisted app settings: `api_keys` — `Apikey.key` is encrypted at rest via `EncryptedString`/`secrets_crypto.py` (Fernet; key from `SECURITY_ENCRYPTION_KEY` or auto-generated at `<data_dir>/.encryption_key`, 0600 — back that file up with the DB, or losing it makes stored keys unrecoverable; legacy plaintext rows still read fine and get encrypted on next save) plus a live quota panel (`routers/quota_routes.py`'s `GET /api/services/quota`, hitting VirusTotal/Shodan/Hunter.io's own quota endpoints with the stored key — the only three wired providers with one), `ai_settings`, `general`, `keywords`, `modules`, `cti_profile` — each has its own models/schemas/crud/routers under `core/settings/<name>/`), `security/ssrf_guard.py` (resolves + validates the hostname of any user-supplied URL before it's fetched server-side, rejecting private/loopback/link-local/reserved IPs; `safe_get(client, url, ...)` is the single shared entrypoint — pins the resolved IP for the request *and* manually re-validates each redirect hop, since httpx's own `follow_redirects` would re-resolve `Location` without going through this check; callers must construct their `httpx.AsyncClient` with `follow_redirects=False`. Covers every feature that fetches a user-supplied or externally-sourced URL over `httpx` (favicon downloads, newsfeed article/feed fetching, LLM-template web content, domain WHOIS/RDAP redirects); opt-out via `SECURITY_ALLOW_PRIVATE_NETWORK_TARGETS`, dev/testing only. `git_recon`'s repo-clone target is the one outbound-fetch path *not* on `httpx` — see its own feature bullet below for the subprocess-argv allowlist that plays the same role there), `reports/` (generic HTML/PDF report renderer — `render_html`/`render_pdf` via Jinja2 + `xhtml2pdf`, `generate_report(title, sections, fmt, locale, ...)`; feature-level `report_service.py` modules build `ReportSection`/`ReportRow` lists with their own small `en`/`ru` label dicts and call into this; used by `ioc_lookup` single-lookup history (`GET /api/ioc-lookup/history/{id}/report`) and `email_analyzer` (`POST /api/email/report`, stateless — takes the analysis result in the body since there's no persisted history to fetch by ID), `alerts/` (WebSocket alerts), `database.py`, `scheduler.py`, `dependencies.py` (includes `get_disk_space_health` — warns at `"low"` below `AppSettings.low_disk_space_threshold_bytes` (env `LOW_DISK_SPACE_THRESHOLD_BYTES`, default 1 GiB) free on `data_dir`; logged once at startup via `main.py`'s `_check_disk_space` and re-checked on every `/api/healthcheck/detailed` call), `exceptions.py`, `healthcheck.py` (basic/detailed/`ready`/`live` probes; detailed status is `"degraded"`, not `"unhealthy"`, when only disk is low).
- `features/` — one directory per product feature, each typically split into `routers/`, `service/`, `schemas/`, `models/`, `crud/`, `utils/`, `config/`:
  - `newsfeed/` — RSS aggregation, IOC extraction from articles, MITRE ATT&CK mapping, trends/analytics. Each `NewsfeedSettings` row tracks `last_fetched_at`/`last_success_at`/`last_error`, updated per-feed by `feed_processing_service.py`'s `process_feed_chunk` on every scheduler run, so a persistently failing feed (bad URL, provider outage) shows a warning in the feed management UI instead of only ever appearing in logs.
  - `ioc_tools/` — `ioc_lookup` (single + bulk lookups against AbuseIPDB, AlienVault, VirusTotal, Shodan, etc.; single-lookup searches are auto-saved to history once every queried service responds, via `POST/GET/DELETE /api/ioc-lookup/history`, persisted as `SingleLookupSearch`/`SingleLookupResult`), `ioc_extractor`, `ioc_defanger`, `domain_finder` (URLScan.io-based typosquat/phishing domain discovery, `/api/domain/lookup`; plus a WHOIS/RDAP panel, `/api/domain/whois`, via the public `rdap.org` bootstrap redirector — no API key, registrar/dates/registrant-org/nameservers only, no historical snapshots since that's a paid tier at every provider).
  - `email_analyzer/` — `.eml` parsing, header/IOC checks, AI-assisted analysis.
  - `image_tools/` — EXIF/metadata extraction, hashing, reverse image search deep-links (no API key needed).
  - `llm_templates/` — user-defined AI prompt templates (categories + templates).
  - `cvss_calculator/` — CVSS 3.1/4.0 scoring.
  - `username_search/` — two pluggable sources behind one unified schema (`MaigretSearch`/`MaigretSiteResult`, with a `source` column: `maigret` | `social_analyzer`, and a `extra` JSON column for source-specific display data), selected per-scan via `ScanRequest.source` (`POST /api/username-search/scan`):
    - **maigret** — called in-process (not subprocess) via `maigret.checking.maigret()`, true per-site progress streamed over SSE; settings under `core/settings/username_search/` (timeout/concurrency/top-sites-count/proxy, site-database auto-update tracking, plus a manual PyPI-latest-version check for the `maigret` package itself — same pattern as social-analyzer below, `POST /api/username-search/maigret/check-update`).
    - **social-analyzer** — invoked as a **subprocess** of its installed CLI (`social_analyzer_service.py`, `--output json`), *not* in-process: the installed package's on-disk module directory is named with a hyphen (not a valid Python identifier), so it can't be imported — only `importlib.metadata` (version/site-count) and the console script are usable. No per-site progress callback exists either, so the SSE stream is coarse (`started` → one terminal event); cancellation kills the subprocess, and a `PROCESS_WATCHDOG_SECONDS` (30 min, `social_analyzer_config.py`) wall-clock ceiling force-kills it even with no client watching, since `timeout_seconds`/`top_sites_count` can both be user-configured to 0 (unbounded). Settings/version-check under `core/settings/username_search/social_analyzer_settings_*` (timeout/top-sites-count, plus a manual PyPI-latest-version check, `POST /api/username-search/social-analyzer/check-update`). For both tools this only flags an available update (`UsernameSearchInfo.update_available`) — installing it still requires a container rebuild, since packages are pinned in `requirements.txt` at image-build time, not runtime-installable by the non-root container user.
    - Report export (`report_service.py`) reuses Maigret's own report writers and only works for maigret-sourced runs; social-analyzer runs just report `has_export: false`.
  - `email_search/` — finds which mail providers a username is registered at, via `mailcat-osint`'s ~26 per-provider checker coroutines (SMTP RCPT probing, provider APIs, registration-form probing, headless Chromium). Driven in-process (no reusable entrypoint exists in mailcat like maigret's, so `service/email_search_service.py` runs the checker coroutines itself, bounded by an `asyncio.Semaphore`), progress streamed over SSE (`POST /api/email-search/scan`), found-provider results persisted (`MailSearch`/`MailSearchResult`). Two checker groups are gated off by default via settings (`core/settings/email_search/`, `enable_smtp_checks`/`enable_headless_checks`): SMTP checkers (Gmail/Yandex/mail.de) need outbound TCP/25 usually blocked in Docker/cloud (workaround: `use_tor`/`proxy_url`, mirroring mailcat's own `--tor`/`--proxy`); headless-Chromium checkers (Fastmail/int.pl/onet.pl, via `requests-html`→`pyppeteer`) lazily download a Chromium binary on first real use. Each checker call is wrapped in a hard per-checker `asyncio.wait_for` timeout; mailcat itself closes its browser handle in a `finally` that survives cancellation, but as a safety net for a wedged Chromium process that never responds to `browser.close()`, `_kill_orphaned_chromium` (psutil-based, diffs child PIDs before/after) reaps any Chromium process still alive after a timeout. Like social-analyzer, version/update-check is a manual PyPI-latest check (`GET /api/email-search/info`, `POST /api/email-search/check-update`) — installed version read from package metadata (no `__version__` attribute on the `mailcat` module itself), latest-available version persisted on `EmailSearchConfig`; installing a newer version still requires a container rebuild.
  - `reddit_search/` — finds a Reddit user's full post/comment history, including moderator-removed and author-deleted content, by querying the public Arctic Shift (`arctic-shift.photon-reddit.com`) and PullPush (`api.pullpush.io`) archive APIs in parallel (no auth/API key; server-side port of the frontend-only [Rosint](https://github.com/zuxu4n/Rosint)). `service/reddit_search_service.py` merges+dedupes results by Reddit item ID and sorts desc by `created_utc`; pagination is timestamp-cursor-based (`before`/`after` on `created_utc`), not offset-based. One `RedditSearch` row persists per username investigation (`POST /api/reddit-search/scan`, called once per page per posts/comments tab), with pages appended idempotently to `RedditSearchResult` (unique on `search_id`+`kind`+`reddit_id`) so re-fetching a page while paging back doesn't duplicate rows. No settings module — no tunables, nothing to configure.
  - `dork_runner/` — runs a static library of parameterized dork templates (`site:`, `filetype:`, `intext:`, etc., `config/dork_templates.py`) against a domain/username/email, ephemeral like `domain_finder` (no persistence/history). DuckDuckGo's HTML endpoint is the default engine since Google/Bing block scripted queries almost immediately; Google/Bing are still selectable but best-effort. Queries run sequentially with a politeness delay between them (`config/search_engines_config.py`), never in parallel. `POST /api/dork-runner/run`, `GET /api/dork-runner/templates`.
  - `git_recon/` — wraps [gitcolombo](https://github.com/Soxoj/gitcolombo) (stdlib-only, imported in-process — `service/git_recon_service.py` drives its `GitAnalyst` directly rather than calling its CLI `main()`, so results come back as structured objects instead of parsed ANSI text) to correlate names/emails/GitHub logins from commit history. Three modes via `POST /api/git-recon/scan` (`ScanRequest.mode`): `search` is API-only (GitHub `/gpg_keys` + `/search/commits`, no cloning); `url`/`nickname` clone one repo or every public non-fork repo of a user/org and correlate author/committer identity mismatches across `git log --all`. gitcolombo's `git clone` is a `subprocess` call that (unlike every other outbound fetch in this codebase) isn't `httpx`, so it's outside `ssrf_guard`'s reach — `validate_github_repo_url`/`validate_github_nickname` in the service module are the sole gate restricting any user-supplied target to `https://github.com/<owner>/<repo>` before it reaches subprocess argv (regression-tested in `tests/features/git_recon/test_url_allowlist.py`, since the repo-wide `test_ssrf_guard_coverage.py` only scans for raw `httpx`/`requests` clients). Clones into an ephemeral `tempfile.mkdtemp()` dir, `rmtree`'d in a `finally`; `config/git_recon_config.py` caps repo count/clone workers/wall-clock time per scan (full, non-shallow clones only — gitcolombo needs `--all` history). Reuses the `github_pat` row from `core/settings/api_keys` rather than a separate credential; unauthenticated GitHub rate limits apply if unset. One `GitReconSearch` row persists per scan, result stored as a JSON blob (graph-shaped output — aliases, same-person clusters — not worth normalizing into FK tables for the only consumer).
- **Access control**: the app has no user accounts (single-user tool) — instead, `core/security/access_control.py` guards every `/api/*` route (plus `/docs`/`/redoc`/`/openapi.json`, added back in `main.py` after `fastapi_config.py` disables FastAPI's built-in ones) behind one shared bearer token, checked via `verify_access_token` wired as a router-level dependency in `router_registry.py`. Token comes from `API_ACCESS_TOKEN` or is auto-generated and persisted to `<data_dir>/.access_token` (0600) on first startup, printed once to stdout/logs — same pattern as the encryption key, factored into `core/security/persisted_secret.py` and shared by both. `healthcheck.router` and the alerts `ws_router` are the only routers registered without this dependency: healthcheck has no sensitive data, and the WebSocket handshake checks the same token itself via a `?token=` query param (browsers can't set a custom header on a WS handshake) — see `alerts_routes.py`. Frontend: `core/services/baseApi.js` attaches the token from `localStorage` (`core/utils/accessToken.js`) to every request and reloads the page on a 401 from an already-set token; `core/components/ui/AccessGate.jsx` shows a one-time entry screen when no token is stored yet, wrapping `AppRoutes` in `App.js`.
- All routers are aggregated in `backend/app/utils/router_registry.py` via `get_core_routers()`, `get_settings_routers()`, `get_feature_routers()`, then mounted in `main.py`.
- Migrations: Alembic, config at `backend/alembic.ini`, scripts in `backend/migrations/versions/`. `docker-entrypoint.py` runs `alembic upgrade head` automatically (as `appuser`, after the data-dir chown) before exec'ing the app, so schema changes apply on every container start with no manual step — a failed migration aborts startup rather than serving the app against a stale schema. Manual runs still work: `docker compose run --rm backend alembic upgrade head`.
- Tests: `backend/tests/`, run with pytest (`pytest.ini` sets `testpaths = tests`, `pythonpath = .`). Currently sparse coverage: `tests/features/image_tools`, `tests/core` (SSRF guard + coverage regression, log redaction).
- CI (`.github/workflows/ci.yml`): besides backend/frontend tests and a docker-build smoke test, runs `pip-audit`/`yarn npm audit` (dependency CVEs) and Trivy (built-image CVEs) as report-only (`continue-on-error: true` — the dependency tree pulls in OSINT tools outside this repo's control, e.g. maigret/social-analyzer/mailcat-osint/pyppeteer, likely already carrying unfixable transitive findings), plus CodeQL (SAST, Python+JS) and gitleaks (secret scanning) as separate blocking jobs. All four are free here (gitleaks-action needs a paid license only for org-owned repos, not personal ones).
- `.github/dependabot.yml` opens weekly update PRs for `pip`, `npm-and-yarn`, `docker` (both Dockerfiles), and `github-actions`, each with a 3-day `cooldown` as a supply-chain mitigation (a malicious release published and yanked within days never reaches a PR).
- `.pre-commit-config.yaml` runs gitleaks locally pre-commit (same tool as CI's `secret-scan` job, catching a secret before it enters git history rather than after a push). Setup: `pip install pre-commit && pre-commit install`.
- `docker-compose.yaml`'s `backend` service has `env_file: {path: .env, required: false}` — a root-level `.env` (see `.env.example`) is NOT picked up by the container otherwise; compose's own automatic `.env` loading only covers variable substitution *within* the compose file, not the container's process environment.

## Frontend architecture

`frontend/src/features/` mirrors the backend feature split — one directory per module (`newsfeed`, `ioc-tools`, `email-analyzer`, `image-tools`, `llm-templates`, `cvss-calculator`, `rule-creator` (Sigma/Yara/Snort rule builder), `username-search`, `email-search`, `reddit-search`, `settings`). Each feature dir typically has its own `components/`, `hooks/`, `services/` (API calls), `constants/`, `utils/`. Entry: `src/App.js` → `src/index.js`. `src/core/` holds shared/cross-feature code.

i18n: `frontend/src/core/i18n/` (`en`/`ru`, one JSON per feature namespace). When batch-translating a feature via subagents, see `frontend/I18N_PATTERN.md` — don't duplicate the code pattern/rules/deliverable format in each subagent's prompt, point at that file instead (only pass scope/namespace/output per subagent).

New top-level features need edits in two places beyond their own directory: `src/core/config/routes.js` (lazy route) and `src/core/config/sidebarConfig.js` (nav entry, plus a tabs config + `Layout.jsx`'s `currentTabs` path-match branch only if the feature has its own sub-tabs — tab-less features like `image-tools`/`dork-runner` skip that and just get top-nav, no sidebar).

Cross-feature "send to X" actions (e.g. `email_search`'s found email → `ioc_lookup`, `ioc_lookup`'s domain result → `domain_finder`) pass the value via a `?q=` query param (`core/utils/crossFeatureNav.js`'s `buildPrefillUrl`); the target feature reads and clears it on mount via `core/hooks/usePrefillFromQuery.js`. `core/components/ui/ChainActionButton.jsx` is the reusable trigger.

## Integrated external services

IOC/threat-intel lookups across IPs, domains, URLs, emails, hashes, CVEs via: AbuseIPDB, AlienVault OTX, CheckPhish, CrowdSec, GitHub, Google Safe Browsing, HIBP, Hunter.io, IPQualityScore, Maltiverse, NIST NVD, Pulsedive, Reddit, Shodan, ThreatFox, Twitter/X, URLScan.io, VirusTotal. ## Conventions worth knowing

- Settings/config uses `pydantic-settings` with per-domain `BaseSettings` subclasses, each with its own `env_prefix` (e.g. `DB_`, `LOG_`, `API_`, `SCHEDULER_`), all loaded from a single `.env` (gitignored, not present in repo — none committed).
- `core/config/log_redaction.py`'s `SecretRedactionFilter` is attached to every logging handler (console + both file handlers, `logging_config.py`) and scrubs API-key-shaped substrings (query params, `Bearer` tokens, IPQualityScore's path-embedded key) out of every log record. Needed because several IOC-lookup providers (Hunter.io, Pulsedive, Shodan, Google Safe Browsing, IPQualityScore) pass their key in the request URL rather than a header, and `httpx.HTTPStatusError`'s `str()` embeds the full URL — a broad `except Exception: logger.error(..., str(e))` would otherwise leak it into `data/logs/*.log`. Matches by pattern (not by looking up live key values), so new providers are covered automatically.
- In route handlers, prefer injecting settings via dependency (`SettingsDep`) rather than importing the module-level `settings` singleton, so tests can override via `app.dependency_overrides`.
- Feature module layering pattern: `routers` (HTTP) → `service` (business logic) → `crud` (DB access) → `models`/`schemas`. New features should follow this same split.
- AGPL-3.0 licensed (versions ≤ v0.1.0 are MIT). Be mindful of this when adding dependencies or discussing licensing.
- Any code that makes an HTTP request to a URL/host supplied (directly or indirectly) by the user must go through `core/security/ssrf_guard.py`'s `safe_get` — otherwise it's an SSRF vector into the docker network, localhost, or cloud metadata endpoints; `tests/core/test_ssrf_guard_coverage.py` fails on a new raw `httpx`/`requests` client that isn't on its reviewed fixed-host allowlist.
- Security headers are set in two independent places, since the backend's middleware never sees the SPA's own response: `core/config/security_config.py`'s `SecurityHeadersMiddleware` covers `/api/*`, `/docs`, `/redoc`, `/openapi.json` (CSP is relaxed to allow `cdn.jsdelivr.net` + inline init script specifically on `/docs`/`/redoc`, since Swagger/ReDoc need it and those routes are access-token gated with no user-supplied content anyway); `frontend/nginx.conf`'s `location /` sets the same base headers + a strict CSP for the actual HTML/JS the browser renders. HSTS stays production-only (`ENVIRONMENT=production`) since it must not be sent over plain HTTP, which is this app's default.
- `backend/requirements.txt` is compiled (pip-compile/`uv pip compile` style, hand-merged since no `requirements.in` is committed) — `maigret` declares `lxml>=6.0.2` but `newspaper4k` pins `lxml<6.0.0`; resolved via `backend/lxml-override.txt` (`lxml==5.4.0`, passed as `--override` in `Dockerfile`) since maigret doesn't actually import lxml directly. Don't bump lxml without re-checking this (`social-analyzer`'s unpinned `lxml` dep and `mailcat-osint`'s `requests-html`→`pyppeteer` chain are both verified compatible with the same override, no additional pin needed).

## Useful commands

- `make up` / `make rebuild` — start (or rebuild + start) full stack via Docker.
- `make up-backend` / `make rebuild-backend`, `make up-frontend` / `make rebuild-frontend` — per-service.
- `docker compose run --rm backend alembic upgrade head` — apply DB migrations.
- Backend tests: `cd backend && pytest`.
- Frontend dev: `cd frontend && yarn start` (uses `--openssl-legacy-provider`); build: `yarn build`; test: `yarn test`.
- App served at `http://localhost:4000` once containers are up.
